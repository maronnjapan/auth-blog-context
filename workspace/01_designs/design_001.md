# 詳細設計書: npm Trusted Publisherで「npm tokenなしpublish」を導入するときに迷う場所をなくす

---

## 1. 書く主題

GitHub Actionsからnpmパッケージをpublishしたい人が、GitHub Secretsに長期npm access tokenを置かずTrusted Publisherへ移行しようとする場面で、初回publishとnpm画面でのTrusted Publisher設定という「完全自動化しにくい手作業」をどう乗り越えるかを扱う。導入の流れ全体を、人間にもブラウザ操作支援AIにも渡せる「半自動ガイド」として整理する。

---

## 2. もっとも伝えたいこと

Trusted Publisherへの移行で本当に詰まるのは仕組みの理解ではなく「初回publish」と「npm画面での手作業設定」であり、そこを完全自動化に逃げず半自動ガイドとして言語化すれば、人間もAIも迷わず進められる。

---

## 3. 読者が読んだ後にいる状態

- 自分のパッケージで初回publishからTrusted Publisher設定、GitHub Actions publishまでを、どの順番でやればよいか迷わず進められる。
- npm tokenをSecretに置かなくてよい理由を自分の言葉で説明でき、「本当に置かなくていいのか」という不安なく移行を判断できる。
- 手作業が残る部分について、完全自動化すべきか半自動ガイドで十分かを、自分の状況で判断できる。

---

## 4. 読者に持ち帰ってほしいこと

- 持ち帰り 1: CIに長期tokenを置く運用は「漏れたら継続的に悪用される資産」を置く運用であり、短命トークンに寄せられるなら寄せるべき、という判断軸。
- 持ち帰り 2: 自動化は「全部自動」か「全部手動」かの二択ではなく、手作業が残る部分は『開くURL・入力値・確認状態を明示する半自動ガイド』に落とすという選択肢がある、という考え方。
- 持ち帰り 3: 手順を人間とAIの両方に渡せる形（URL・入力値・確認ポイントの明示）で書いておくと、ブラウザ操作支援AIに画面操作を任せやすくなる、という設計指針。

---

## 5. 詳細目次

### 第1章 なぜTrusted Publisherに移行したいのか

#### 1-1. はじめに：やりたいことと、避けたいこと

**節内で書くこと:**
- npmパッケージをGitHub Actionsからpublishしたい、というゴールを最初に置く。
- ただしGitHub Secretsに長期npm access tokenを置き続けるのは避けたい、という制約を並べる。
- この記事が「概念説明」ではなく「初回publishからActions publishまでを実装ベースで通す」記事であることを宣言する。
- 読者が「どこまで自分でやることになるのか」を冒頭で把握できるよう、ゴール状態を1段落で示す。
- 想定読者の前提を明示する。npmでのinstallは経験済み、GitHub Actionsの存在と簡単な構文は知っている層を想定する。npm publishやTrusted Publisherは初めてでもよい、という線引きにする。

**著者の熱量が入る場所:**
- 「Trusted Publisherという言葉は知っているが、公式ドキュメントだけでは導入の流れがイメージしづらい」という、著者自身が感じた入口の難しさ。

#### 1-2. 静的npm tokenをCIに置くリスク

**節内で書くこと:**
- npm access tokenとは何か（publish権限を持つ資格情報であること）を短く確認する。
- CI/CDのSecretに長期tokenを置くと何がリスクになるかを書く。具体的には、漏洩時にtokenの有効期間中ずっと悪用され得る点。
- token漏洩時に起こり得ること（悪意あるバージョンのpublishなど）を失敗シナリオとして1つ挙げる。
- 上記論拠となりそうなニュースのリンクと概要を記載する。
- だから「短命トークンで都度publishしたい」という動機につなげる。

**著者の熱量が入る場所:**
- 「新しい仕組みだから使う」のではなく「長期tokenをCIに置く運用そのものが不安だから移行したい」という、動機の順序へのこだわり。

#### 1-3. Trusted Publisherとは何か

**節内で書くこと:**
- Trusted Publisherの概要を、GitHub Actionsとnpmの間に信頼関係を結ぶ仕組みとして説明する。
- CI実行時に短命トークンを使ってpublishする考え方を書く。
- 静的npm token方式との違いを対比で示す（事前にtokenを保存する／実行時にtokenを得る）。
- npm tokenをGitHub Secretsに置かなくてよい理由を、ここで読者が手順前に納得できる粒度で書く。

**著者の熱量が入る場所:**
- 読者が「手順だけなぞる」のではなく「なぜtokenなしで成立するのか」を理解した状態で先に進めるようにしたい、という意図。

**節内で書くこと:**

- それぞれが記事のどのパートに対応するかを先に地図として示す。

**著者の熱量が入る場所:**
- （特になし。読者の現在地を示す案内パート）

#### 2-2. 完全自動化ではなく「半自動ガイド」にする理由

**節内で書くこと:**
- npm画面でのTrusted Publisher設定はWeb上の手作業で、CLIだけでは完結しないという事実を書く。
- これをPlaywright等で完全自動化すると重くなる、という判断を書く。
- だから手作業は残しつつ、開くURL・入力する値・確認すべき状態を明示する「半自動ガイド」に倒す、という方針を宣言する。
- これがこの記事の独自性であることを明示する。

**著者の熱量が入る場所:**
- 「完全自動化を目指して重くするより、迷わない手順に言語化する方が実用的」という、著者の方針そのもの。ここが記事の核。

#### 2-3. 人間にもAIにも渡しやすい形にする

**節内で書くこと:**
- 半自動ガイドを、人間が読むだけでなくClaude in ChromeやChatGPT Agentなどのブラウザ操作支援AIにも渡せる形にする狙いを書く。
- そのために守る書き方を列挙する。開くURLを明示／入力値を明示／クリック箇所を説明／設定後の確認状態を明示。
- publishや権限変更など重要操作の前は人間が確認する、という線引きを書く。
- AIに完全委任せず「画面遷移と入力補助を任せる」位置づけにする、という設計思想を書く。

**著者の熱量が入る場所:**
- ブラウザ操作支援AIが現実的な選択肢になった今だからこそ、手順を「AIにも渡せる構造」で書いておきたい、という時流への反応。

### 第3章 初回publishを通す

#### 3-1. 初回publish前に確認すること

**節内で書くこと:**
- Trusted Publisherを設定する前提として、パッケージが一度publishされている必要があることを書く。
- package.jsonの確認ポイントを書く。package name、version。
- npm loginが済んでいるかの確認方法を書く。
- ここでつまずくとTrusted Publisher設定までたどり着けない、という位置づけを明示する。

**著者の熱量が入る場所:**
- 「初回publishで止まると、その先に進めない」という、読者が最初に詰まりやすい箇所への先回り。

#### 3-2. 初回publishを実行し、結果を確認する

**節内で書くこと:**

- publish後にnpmパッケージページで公開を確認する手順を書く。
- 次のTrusted Publisher設定への導線を示す。

**著者の熱量が入る場所:**
- （特になし。手順の実行パート）

#### 3-3. 初回publishガイドスクリプトという形にする

**節内で書くこと:**
- コマンドの羅列ではなく、CLI上で確認と操作を順番に案内するガイドスクリプトにする狙いを書く。
- スクリプトが案内する流れを列挙する。package name表示／version表示／npm login済みかの確認／publish前チェック／publishコマンド案内／publish後のパッケージページ表示／Trusted Publisher設定への誘導。
- 「手を動かしやすい形」にすることが目的であると明示する。
- ガイドスクリプトは実コードとして本文に載せる。ただし分量で本筋が埋もれないよう、デフォルトは折りたたみ（toggle/details）で隠し、読みたい読者だけ展開できる形にする。

**著者の熱量が入る場所:**
- 「コマンドだけ書いても読者は手が止まる。確認と次の一手まで案内したい」という、読者の手の動きへの配慮。

### 第4章 Trusted Publisherを設定する（半自動ガイドの本体）

#### 4-1. npm画面でTrusted Publisherを設定する手順

**節内で書くこと:**
- npmのパッケージ設定ページを開くURLを明示する。
- Trusted Publisher設定画面への移動を案内する。
- 入力する値を明示する。GitHub owner / repository、workflow file名。
- 設定完了後に確認すべき状態を書く。
- npm側のUI（メニュー名・項目名）は変わり得るため「執筆時点の画面に基づく」旨を短く注記する。ただし注記は最小限にとどめ、今すぐ動かせることを優先する。

**著者の熱量が入る場所:**
- 「Trusted Publisherはnpmのどこで設定するのか」という、著者自身が最初に迷ったポイント。

#### 4-2. 設定値をCLIで表示して、画面に転記しやすくする

**節内で書くこと:**
- npm画面で入力する値（owner / repository / workflow file名）を、CLI側で表示して転記しやすくする工夫を書く。
- 人間が見ても進められ、AIに渡しても入力補助を任せられる形になることを書く。
- 実際にAIが操作できるプロンプトも出力されるようにし、そのプロンプト例を示す。
- 重要操作（設定確定）の前に人間が確認する線引きを再掲する。

**著者の熱量が入る場所:**
- 「画面とCLIを行き来する手間を、表示で吸収したい」という、導入時の摩擦を減らす工夫。
