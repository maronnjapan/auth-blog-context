# PKCE の `code_verifier` はなぜ 43〜128 文字なのか

## メモの目的

PKCE（Proof Key for Code Exchange）を調べていると、`code_verifier` に **43〜128文字** という長さ制約がある。

最初に見たときは、

「ランダムな文字列なら、別に自由な長さでよくない？」
「なぜ 43 文字以上なの？」
「なぜ 128 文字以下なの？」
「S256 の `code_challenge` はいつも 43 文字っぽいけど、それと関係あるの？」

という疑問が出てくる。

この記事では、そのあたりを RFC 7636 を読みながら整理する。

ただし、現時点では自分自身もまだ完全に腹落ちしていない部分がある。

特に、

- なぜ 256 bit のエントロピーが基準として置かれているのか
- なぜ上限が 128 文字なのか
- 仕様策定者がどこまで明確な意図を持ってこの数字を選んだのか

については、まだ追加調査が必要。

なのでこの記事では、まず RFC に書かれている事実と、そこから読み取れそうな考察を分けて整理する。

---

## この記事で書きたいこと

この記事の主題は、PKCE の `code_verifier` がなぜ 43〜128 文字なのかを整理すること。

特に、以下のような流れにしたい。

1. PKCE とは何かを簡単に説明する
2. `code_verifier` と `code_challenge` の関係を説明する
3. RFC 7636 上の文字数制約を確認する
4. 下限 43 文字の理由を考える
5. 上限 128 文字の理由を考える
6. まだ自分が理解しきれていない点を明記する
7. 実装する場合はどう考えればよいかを整理する

---

## 前提：PKCE とは何か

PKCE は、OAuth 2.0 の Authorization Code Flow において、認可コードを横取りされた場合でも、攻撃者がアクセストークンを取得しにくくするための仕組み。

ざっくり言うと、クライアントは認可リクエストの前にランダムな秘密値を作る。

このランダムな秘密値が `code_verifier`。

そして、その `code_verifier` から `code_challenge` を作る。

認可リクエストでは `code_challenge` を送る。

その後、トークンリクエストでは元の `code_verifier` を送る。

認可サーバーは、受け取った `code_verifier` から改めて `code_challenge` を計算し、最初の認可リクエストで受け取っていた `code_challenge` と一致するかを確認する。

これにより、認可コードだけを盗まれても、元の `code_verifier` が分からなければトークン交換できない、という構造になる。

---

## RFC 7636 における `code_verifier` の定義

RFC 7636 では、`code_verifier` は以下のように定義されている。

```text
code-verifier = 43*128unreserved

unreserved = ALPHA / DIGIT / "-" / "." / "_" / "~"
````

つまり、`code_verifier` は以下の条件を満たす必要がある。

* 43 文字以上
* 128 文字以下
* 使用できる文字は URL セーフな unreserved 文字

  * A-Z
  * a-z
  * 0-9
  * *
  * .
  * _
  * ~

この時点でまず疑問になるのが、なぜ 43〜128 文字なのかという点。

---

## `code_challenge` の定義

PKCE では、`code_challenge_method` として `plain` と `S256` がある。

`plain` の場合は、単純に以下のようになる。

```text
code_challenge = code_verifier
```

一方、`S256` の場合は以下のように定義されている。

```text
code_challenge = BASE64URL-ENCODE(SHA256(ASCII(code_verifier)))
```

ここで重要なのは、`S256` の場合、`code_challenge` は SHA-256 の結果から作られるということ。

SHA-256 の出力は常に 256 bit、つまり 32 byte。

その 32 byte を base64url エンコードし、padding の `=` を省略すると、結果は 43 文字になる。

つまり、`S256` の `code_challenge` は、元の `code_verifier` が 43 文字でも 128 文字でも、基本的に 43 文字になる。

---

## ここで混乱しやすいこと

`code_verifier` と `code_challenge` の文字数は、同じように見えて、実は意味が違う。

`code_verifier` はクライアントが生成するランダムな秘密値。

一方、`S256` の `code_challenge` は、その `code_verifier` を SHA-256 にかけた結果を base64url エンコードした値。

そのため、`S256` の `code_challenge` が 43 文字になるのは、SHA-256 の出力サイズが 32 byte で固定だから。

`code_verifier` の下限が 43 文字であることと、`S256` の `code_challenge` が 43 文字になることは、関係はあるが、同じ話として雑にまとめると混乱しそう。

ここは記事内でも丁寧に分けて説明したい。

---

## 下限 43 文字について

RFC 7636 では、`code_verifier` の生成方法として、32 オクテットのランダム値を作り、それを base64url エンコードする方法が書かれている。

32 オクテットは 32 byte。

1 byte は 8 bit なので、

```text
32 byte × 8 bit = 256 bit
```

になる。

この 32 byte のランダム値を base64url エンコードし、padding を省くと、文字数は 43 文字になる。

つまり、43 文字という下限は、32 byte、つまり 256 bit のランダム値を base64url で表現したときの長さと対応している。

---

## 43 文字の計算イメージ

base64 は、3 byte を 4 文字に変換する。

32 byte を base64 エンコードする場合を考える。

まず、30 byte は 3 byte のまとまりが 10 個なので、40 文字になる。

```text
30 byte -> 40 文字
```

残りは 2 byte。

base64 では、残り 2 byte は 3 文字 + padding 1 文字になる。

```text
2 byte -> 3 文字 + "="
```

なので、32 byte 全体では通常の base64 だと 44 文字になる。

```text
32 byte -> 44 文字
```

ただし、PKCE では base64url encoding で、padding の `=` を省く。

そのため、最後の `=` が消えて 43 文字になる。

```text
32 byte -> 44 文字
padding 1 文字を除去
= 43 文字
```

これが、32 byte のランダム値を base64url without padding で表現すると 43 文字になる理由。

---

## ただし、ここでまだ理解しきれていないこと

ここまでは、計算としては理解できる。

32 byte のランダム値を base64url エンコードすると 43 文字になる。

だから、43 文字という数字が出てくる。

ただし、まだ自分の中で完全に理解できていないのは、

「そもそも、なぜ 256 bit のランダム値が基準なのか」

という点。

RFC 7636 には、`code_verifier` は暗号学的にランダムであり、推測困難であるべきという趣旨が書かれている。

また、32 オクテットの乱数を使う生成方法も示されている。

しかし、「なぜ 128 bit ではなく 256 bit なのか」「なぜこの用途で 256 bit を選んだのか」について、自分はまだ明確に説明できていない。

考えられる理由としては、

* SHA-256 と同じ 256 bit に揃えている
* 十分に高いエントロピーを持たせるため
* 将来的な安全性も含めた余裕を持たせるため
* OAuth の認可コード横取り攻撃に対して、総当たりが現実的に不可能な水準にするため
* 乱数生成の推奨値として 32 byte が扱いやすかったため

などが考えられる。

ただし、これは現時点では自分の推測を含む。

記事にするなら、ここは断定せず、

「43 文字という下限は、RFC が推奨する 32 オクテットのランダム値を base64url エンコードした長さと対応している。ただし、なぜ 256 bit を基準にしたのかについては、追加で仕様策定時の議論やセキュリティ上の背景を確認したい」

という書き方にしたい。

---

## 42 文字ではだめなのか

base64url の文字は、おおむね 1 文字あたり 6 bit の情報を表現できる。

単純計算すると、42 文字では以下になる。

```text
42 文字 × 6 bit = 252 bit
```

256 bit には少し届かない。

43 文字なら以下になる。

```text
43 文字 × 6 bit = 258 bit
```

256 bit を表現するには 43 文字が必要になる。

ただし、この説明も少し注意が必要。

これは「base64url 文字列として 256 bit 相当を表現するには 43 文字必要」という説明であって、「43 文字の任意の文字列が必ず 256 bit のエントロピーを持つ」という意味ではない。

エントロピーは文字数だけではなく、どのようにランダムに生成したかに依存する。

例えば、43 文字あっても、人間が考えた文字列や、偏りのある疑似乱数で作られた文字列なら、十分なエントロピーがあるとは限らない。

そのため、実装上は「43 文字以上にする」だけでは不十分で、暗号学的に安全な乱数生成器を使う必要がある。

---

## 上限 128 文字について

上限の 128 文字については、下限の 43 文字ほど明確に説明しにくい。

下限 43 文字は、32 byte のランダム値を base64url エンコードすると 43 文字になる、という形で説明しやすい。

一方で、128 文字という上限については、RFC 7636 だけを読んでも、自分はまだ明確な理由を説明できていない。

考えられる理由としては、以下がある。

* 無制限に長い入力を許さないため
* 実装上、扱う文字列長を現実的な範囲に制限するため
* 認可リクエストやトークンリクエストのパラメータ長を過剰にしないため
* `plain` の場合、`code_challenge = code_verifier` になるため、`code_challenge` 側にも現実的な上限が必要だったため
* 43 文字以上あれば十分な安全性があり、それ以上は過度に長くする必要がないため

ただし、これらは現時点では考察・推測であり、RFC 上で明確にそう説明されているとまでは言えない。

記事にする場合は、ここを断定しない。

例えば、以下のように書くのがよさそう。

「下限 43 文字は、32 オクテットのランダム値を base64url エンコードした長さとして説明しやすい。一方で、上限 128 文字については、RFC 7636 の本文だけから明確な意図を断定するのは難しい。実装上の現実的な上限として設けられたものと考えられるが、この点は仕様策定時の議論などを追加で確認したい。」

---

## `plain` があることによる影響

`S256` の場合、`code_challenge` は SHA-256 の出力を base64url エンコードするため、基本的に 43 文字になる。

しかし、PKCE には `plain` という方式もある。

`plain` の場合は、

```text
code_challenge = code_verifier
```

になる。

つまり、`code_verifier` が 43〜128 文字なら、`code_challenge` も 43〜128 文字になる。

RFC の ABNF では、`code_challenge` も `43*128unreserved` と定義されている。

これは、`plain` の場合も考えると自然に見える。

ただし、現在の実装では基本的に `S256` を使うべきであり、`plain` は後方互換や制約のある環境向けと考えたほうがよさそう。

---

## 実装時にはどうすればよいか

実装する側としては、基本的には RFC に書かれている推奨に従えばよい。

つまり、

1. 暗号学的に安全な乱数生成器を使う
2. 32 byte のランダム値を生成する
3. それを base64url エンコードする
4. padding の `=` を取り除く
5. 43 文字の `code_verifier` として使う
6. `S256` で `code_challenge` を生成する

という流れにする。

この方法なら、仕様上の 43〜128 文字の範囲に収まり、かつ RFC が示す推奨方法にも沿う。

無理に 64 文字や 128 文字の `code_verifier` を作る必要はあまりなさそう。

もちろん、43〜128 文字の範囲で、十分なランダム性があるなら仕様上は問題ない。

ただし、記事では「文字数が長ければ安全」というより、「暗号学的に安全な方法で生成された十分なエントロピーを持つ値であること」が重要だと書きたい。

## 現時点での理解まとめ

現時点で、自分の理解は以下。

### 確からしいこと

* RFC 7636 では `code_verifier` は 43〜128 文字と定義されている
* 使用できる文字は unreserved 文字
* RFC では、32 オクテットのランダム値を base64url エンコードする生成方法が示されている
* 32 オクテットは 256 bit
* 32 byte を base64url without padding で表現すると 43 文字になる
* そのため、下限 43 文字は 256 bit のランダム値を base64url 表現した長さと対応している
* `S256` の `code_challenge` は SHA-256 の出力を base64url エンコードするため、基本的に 43 文字になる

### まだ理解しきれていないこと

* なぜ 256 bit が基準として選ばれたのか
* なぜ上限が 128 文字なのか
* 仕様策定者が 43〜128 文字という範囲にした明確な意図
* 当時のメーリングリストや議論で、この文字数について説明されているかどうか

### 推測として書けそうなこと

* 43 文字は、32 byte / 256 bit のランダム値を base64url で表現するための自然な下限
* 128 文字は、実装上・運用上の現実的な上限として設けられていそう
* `plain` メソッドでは `code_challenge = code_verifier` になるため、`code_challenge` 側にも 43〜128 文字の幅が必要だった可能性がある
* ただし、これらは断定しない

## 現時点の結論

現時点では、以下のように整理できる。

`code_verifier` の下限 43 文字は、RFC 7636 が推奨する 32 オクテットのランダム値を base64url without padding で表現した長さと対応している。

32 オクテットは 256 bit であり、PKCE では推測困難な `code_verifier` が必要になるため、この程度のエントロピーを持つランダム値を使うことが意図されていると考えられる。

ただし、なぜ 256 bit が基準として選ばれたのか、また上限が 128 文字である理由については、RFC 本文だけではまだ自分には明確に説明しきれていない。

そのため、記事ではこの点を推測として扱い、追加調査ポイントとして明記する。
