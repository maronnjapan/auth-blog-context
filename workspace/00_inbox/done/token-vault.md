# Slide 6. Token Vault の概要

## スライドに載せる内容

```text id="9be38t"
Token Vault とは

Auth0ユーザー
   ↓ 紐づけ
Connected Account
   ↓
External Provider
Google / Slack / GitHub / Microsoft / Salesforce ...

Auth0 が外部プロバイダーの
access token / refresh token を保存・管理

アプリケーションは必要なときに
外部API用 access token を取得
```

## 話す内容

```text id="r9qsq2"
Token Vault の基本的な考え方は、
Auth0ユーザーと外部サービスのアカウントを紐づけることです。

たとえば、Auth0上のユーザーAが、
GoogleアカウントやSlackアカウントを連携します。

その連携が完了すると、
Auth0 は外部プロバイダーから受け取ったアクセストークンやリフレッシュトークンを
Token Vault に保存します。

アプリケーションは、その保存された外部トークンを直接管理するのではなく、
Auth0 のトークンを使って Token Vault に問い合わせ、
外部APIを呼ぶためのアクセストークンを取得します。

ここで重要なのは、
Auth0のトークンが何もないところからGoogleやSlackのトークンに魔法のように変換されるわけではない、という点です。

事前にユーザーが外部サービス連携に同意していて、
その結果として保存された外部トークンを、
Auth0経由で安全に取り出す、という理解が正確です。
```

# Slide 14. トークンの安全な保存

## スライドに載せる内容

```text id="l38si4"
4. トークンの安全な保存

- access token / refresh token を平文保存しない
- KMS等で暗号化鍵を管理
- DynamoDB / RDS などに暗号化して保存
- provider client secret は Secrets Manager 等で管理
- token 値をログに出さない
```

## 話す内容

```text id="et9d5t"
次に、トークンの安全な保存です。

特にリフレッシュトークンは、
外部APIへの継続的なアクセス権そのものです。

これを普通のアプリケーションデータと同じ感覚で平文保存するのは危険です。

AWSで自作するなら、
KMSなどで暗号化鍵を管理し、
DynamoDBやRDSに暗号化したトークンを保存する構成が考えられます。

また、プロバイダーごとの client secret は Secrets Manager などで管理する必要があります。

さらに、ログにアクセストークンやリフレッシュトークンそのものを出さない設計も必要です。

障害調査のためにログを出した結果、
トークンがログに残ってしまう、というのは避ける必要があります。
```

```text id="0tzyz6"
6. refresh token による更新

access token 期限切れ
    ↓
refresh token を復号
    ↓
provider の token endpoint に request
    ↓
新しい access token を取得
    ↓
必要に応じて refresh token も更新
    ↓
保存情報を更新して返却
```

## 話す内容

```text id="vq80j0"
アクセストークンには有効期限があります。

そのため、
保存済みのアクセストークンをそのまま返すだけでは不十分です。

期限切れ、または期限切れが近い場合には、
保存しているリフレッシュトークンを使って、
外部プロバイダーのtoken endpointにリクエストし、
新しいアクセストークンを取得する必要があります。

プロバイダーによっては、
リフレッシュ時に新しいリフレッシュトークンが返ってくる場合もあります。

その場合、古い情報を更新して保存し直す必要があります。

また、リフレッシュトークンが失効していたり、
ユーザーが外部サービス側で連携解除している場合には、
再同意が必要な状態として扱う必要があります。
```

# Slide 24. スコープの粒度に関する懸念

## スライドに載せる内容

```text id="ih6ss6"
連携時に許可したscope
    ≠
実行時に必要な最小権限
```

## 話す内容

```text id="9ag2im"
もう1つの懸念は、スコープの粒度です。

外部サービス連携時には、
ある程度まとまったスコープをユーザーに許可してもらうことがあります。

しかし、実際の処理単位で見ると、

別の処理では予定作成が必要かもしれません。

つまり、
連携時に許可したスコープと、
実行時に必要な最小権限の粒度が一致しない可能性があります。

これはAIエージェントの文脈でも重要ですが、
AIエージェント以外の外部API連携でも同じように重要です。
```
