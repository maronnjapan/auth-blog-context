- はじめに
    - 少し前にAuth0が[Token Vault](https://auth0.com/docs/ja-jp/secure/tokens/token-vault)を発表した。
    - Token Vaultを見つけた経緯がAI Agentでの文脈だったので、AI専用の機能かとおもっていた。
    - しかし、実際に触ってみるともちろんAIの文脈で使えることは間違いないが、必ずしもAIと密結合ではないことが分かった。
    - Auth0発行のトークンと外部プロバイダーのトークン交換を行ってくれる便利なストアであることが分かった。
    - なので、この記事ではToken Vaultの概要であったり、ユースケース、使ってみた感想を記載する。
    - 自分で試すことができるリポジトリも作ったので、活用してほしい。
- Token Vaultとは
  - Token Vaultはトークン専用のシークレットストアである。
  - ただし、一般的なシークレットストアとは役割が異なる。
    - 一般的なシークレットストアは、シークレットの安全な保管・バージョニング・ローテーションなど「安全な保存」に主眼がある。
    - 一方Token Vaultは、安全に保存することに加え「トークン交換（Token Exchange）」を前提とした機能を持つ。
  - Auth0で認証されたユーザーに対して発行されるトークンを起点として、外部プロバイダー用のトークンを取得する仕組みを提供する。
  - 主にリフレッシュトークンを起点として外部プロバイダー用トークンを取得する。
    - Auth0が発行するリフレッシュトークンをToken Vaultに渡すことで、外部プロバイダーのトークンを取得できる。
  - ただしSPAなどリフレッシュトークンの保持が難しいケースでは、アクセストークンを使用するパターンもある。
  - 一度外部プロバイダーとの連携を行えば、その後はリフレッシュトークンを用いてトークン取得が可能になるため、都度ユーザーに再認証を求める必要はない。
  - 本記事では概念理解を優先しているため、正確な仕様については公式ドキュメントを参照すること。
    - https://auth0.com/ai/docs/intro/token-vault#what-is-token-vault

- Token Vaultのユースケース
  - 基本的に「Auth0で認証されたアプリケーションが外部APIを利用する場合」に活用できる仕組みだと考えられる。

  - AIエージェント
    - 近年最も分かりやすいユースケースの一つ。
    - AIエージェントは自律的に動作し、外部APIを呼び出すケースが多い。
    - そのたびにユーザーに認証を求めるフローはUX的に現実的ではない。
    - Token Vaultを利用すると、一度外部プロバイダーとの連携を行えば、リフレッシュトークンを用いて外部トークンの取得が可能になる。
    - その結果、人間の操作を減らしつつセキュリティを担保したAPI呼び出しが実現できる。
    - Auth0がAI Agent向けドキュメントを提供しているのも、このユースケースを強く意識しているためだと思われる。

  - API Gatewayを用いた外部API統合
    - 複数のSaaSや外部APIを統合的に扱うシステムでは、API Gatewayを用意するケースがある。
    - クライアントやバックエンドは、そのAPI Gatewayに対してリクエストを送る。
    - API GatewayはAuth0のトークンを受け取り、それを元にToken Vaultを利用して外部プロバイダーのトークンを取得する。
    - 例えば以下のような構成が考えられる。
      - リクエストに「利用するプロバイダー（Google / GitHubなど）」の情報が含まれる
      - API GatewayがToken Vaultを利用して該当プロバイダーのトークンを取得
      - 外部APIを呼び出し、取得したデータをレスポンスとして返す
    - この構成では、呼び出し元はAPI Gatewayを叩くだけで外部サービスのデータを取得できる。
    - 外部APIごとの認証処理やトークン取得ロジックを各サービス側で実装する必要がなくなる。
- アプリのセットアップ
    - https://github.com/maronnjapan/sample-id-app/tree/auth0-token-vaultのREADME.mdを参照して、セットアップをして。
    - ただ、Auth0,Auth0 CLI,Cloudflare、wranglerコマンド,terraformコマンドが設定できていればスクリプト実行で設定できる。（GoogleのOAuthクライアントは作成が必要）
- Token Vaultを触った所感
  - Cross App Accessの簡易版という印象
    - Cross App Accessにおけるトークン交換をIdP起因で行うという思想は似ている。
    - 必要なトークン取得処理をシステム側が肩代わりするという点でも共通点がある。
    - AIエージェントなど、システム間通信が増えている現状に対応した仕組みだと感じる。

  - ただしCross App Accessと同一視するのは危険
    - 実行するサービスごとに細かく権限を分離しながらトークンを交換する、といった用途は難しそう。
    - そのため思想が似ている部分はあるが、かなり似た仕組みとして捉えるのは危険。
    - そもそも公式ドキュメントでも、Token VaultはOAuth 2.0 Token Exchange (RFC 8693)をベースにしていると説明されている。
      - https://auth0.com/ai/docs/intro/token-vault#what-is-token-vault

  - 認証の代替ではない
    - Token Vaultの設定ではGoogleアカウントなど外部プロバイダーとの連携を行う。
    - またフローを見るとAuthorization Code Flowに似ている部分もある。
    - そのため「認証の代替として使えるのでは？」と感じる可能性がある。
    - しかし実際にはToken Vaultを利用するためにはAuth0のトークンが必要になる。
    - そのためToken Vaultは認証の代替ではなく、「外部トークン取得の仕組み」である。
